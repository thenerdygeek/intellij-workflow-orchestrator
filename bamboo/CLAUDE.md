# :bamboo Module

Build monitoring, CI feedback, CVE remediation. PR display only — creation lives in `:pullrequest`.

## PrBar (Build-tab PR strip)

`PrBar` is a **passive mirror** of the PR tab's selection (redesigned 2026-04-27). It does not fetch PRs from Bitbucket itself, does not poll, and has no Create PR button. It subscribes to `WorkflowContextService.state` and renders:

- **PR-rendered state** (`interactionMode == Live` AND `focusPr != null` — i.e., the focused PR's source branch matches the local checkout): single `PR #id  fromBranch → toBranch` strip with "Open in browser" link.
- **Empty state** otherwise: blue strip with *"No PR for this branch — pick one in the PR tab →"* (clickable, switches to the PR tab).

PR title is intentionally not rendered (`PrRef` doesn't carry it; re-introducing a Bitbucket fetch would defeat the redesign). To restore titles, extend `PrRef.title` from the PR tab side rather than fetching in PrBar.

## Build-details branch-match guard

When `WorkflowContext.interactionMode == ReadOnly` (focused PR's branch ≠ local checkout), the action toolbar's **Refresh**, **Trigger Build**, **Trigger Manual Stage** buttons are disabled with a tooltip identifying the focused PR's branch. The build list, jobs panel, and log viewer stay visible and readable. Implemented via `BuildDashboardActionGate.isLiveMode(ctx)` + a `WorkflowContextService.interactionModeFlow` collector that calls `cachedActionToolbar.updateActionsImmediately()` on transitions.

## PR creation entry point

PR creation only exists on the PR tab now. `:bamboo` no longer depends on `:pullrequest` (delegation happens via the `createPrLauncher` extension point in `:core`); the EP signature is `launch(project, scope)` — the `autoGenerateDescription` parameter was removed in the same redesign.

`PrDescriptionGenerator` and `MarkdownToHtml` live in `:pullrequest/service/`.

## Bamboo Server REST API

Auth: `Authorization: Bearer <PAT>`
Base: `https://{host}/rest/api/latest/`

Key endpoints:
- `GET /rest/api/latest/currentUser` — test connection
- `GET /rest/api/latest/result/{planKey}/latest` — latest build result
- `GET /rest/api/latest/result/{buildKey}` — specific build result + stages
- `GET /rest/api/latest/result/{buildKey}/log` — build log
- `POST /rest/api/latest/queue/{planKey}` — trigger build (with variables)
- `GET /rest/api/latest/result/{planKey}` — running/queued builds. **COLLECTION-EXPAND TRAP (fixed 2026-06-06):** this is a *collection* endpoint — the `expand` MUST be prefixed `results.result.` (i.e. `expand=results.result.stages.stage.results.result`), exactly like `getRecentResults`. The bare `stages.stage.results.result` expands stages inside a result that was never expanded, so Bamboo returns only the `results:{size:N}` wrapper with an empty `result[]` array and the client reports "no running builds" (probe-confirmed on real DC 10.2.14: `result_running_queued.json` had `size:5` but no array; `result_recent.json` with the prefix had 10 populated entries). `getLatestResult`/`getBuildResult` are *single-result* endpoints (`/result/{key}/latest`, `/result/{resultKey}`) where the bare `stages.stage.results.result` is correct — there's no top-level collection to prefix.
- `POST /rest/api/latest/plan/{branchPlanKey}/enable` — enable a (disabled) plan branch. A branch IS a plan, so enable its own key. `BambooApiClient.enablePlanBranch` → `postForm` (sets `X-Atlassian-Token: no-check`), surfaced as `BambooService.enablePlanBranch(branchPlanKey): ToolResult<Unit>`. ⚠ NOT probed against live DC 10.2.14 (added 2026-05-26); non-admin PATs likely get 403 (mapped to a plan-admin permission hint). Verify on a live server before relying on it.

Build variables include `dockerTagsAsJson` — JSON payload of service-to-docker-tag mappings used by automation suites.

## Architecture

- `BambooApiClient` — HTTP client for all Bamboo REST calls. The shared `paginate()` loop (used by `getPlans`/`getBranches`/`getPlanBranches`/`getLinkedRepositories`) advances the offset from the **requested** start-index plus the **actual item count returned** — never from the server-echoed `start-index`/`size` fields. Bamboo's `/plan/{key}/branch` endpoint echoes `start-index: 0` on every page, so trusting it stalled the cursor and re-fetched the same window up to the page cap, duplicating branches in the Automation branch selector (fixed 2026-05-26; regression test in `BambooApiClientPaginationTest`). `BambooServiceImpl.getPlanBranches` also `distinctBy { it.key }` as a defensive second layer.
- `BambooServiceImpl` — implements `BambooService` (in :core), returns `ToolResult<T>`
- `BambooServiceImpl` returns `BuildResultData.stages[].jobs[]` populated from `?expand=stages.stage.results.result` responses. Each `BuildJobData.resultKey` (e.g. `PROJ-PLAN138-UNIT-4`) is what the agent passes to `bamboo_builds.get_build_log` for per-job logs.
- `BuildPlanResolutionPolicy` (in `:bamboo/ui/`) is a pure object encapsulating the Build dashboard's "use detected plan / use configured master / show hint" decision, so the resolution matrix is testable without IntelliJ infrastructure. The dashboard delegates to it after every `BambooService.autoDetectPlan` call.
- `BuildMonitorService` — background polling via SmartPoller, emits `BuildFinished`/`BuildLogReady` events. Uses platform-injected `cs: CoroutineScope` (see `:core` "Service & threading conventions"). `PrBar` git reads use `readAction { }`.
  - **T-B2/B3-a (2026-05-11):** `BuildMonitorService` now subscribes to `WorkflowContextService.state.map { it.focusBuild }.distinctUntilChanged()` via an internal `wireFocusBuildSubscription` method wired from the IntelliJ-DI constructor. Non-null `focusBuild` → `startPolling(chainKey ?: planKey, branch, interval)`; null → `stopPolling()`. Uses `collectLatest` so rapid focus changes cancel the in-flight body and only the latest target wins. The test constructor accepts an optional `focusBuildFlow: StateFlow<BuildRef?>?` to inject a test-controlled flow without IntelliJ DI. `startPolling` and `stopPolling` are `open` to allow recording subclasses in tests.
  - **T-B2/B3-b (2026-05-11):** Polling lifecycle migration completed. `BuildMonitorStartupActivity` deleted (was polling the wrong plan in multi-repo setups — bug B2). `BuildDashboardPanel` no longer calls `startPolling`, `switchBranch`, or `stopPolling` directly — all three were removed. `dispose()` no longer stops polling (ambient polling continues for Automation + Handover consumers). The "select a branch from dropdown" code path (formerly `switchBranch` at lines 265, 275) now routes through `reroutableFocusPr(branchName)` → `WorkflowContextService.focusPr(matchedPr)` when a matching open PR exists, or logs and skips when no PR matches (local override: branch has no open PR — ambient polling unchanged). The PR-lookup uses `BuildDashboardPanel.findMatchingPrForBranch(openPrs, branch, preferredRepoName)` — a pure companion-object function testable without IntelliJ infra, exercised in `BuildDashboardPanelDropdownTest`. `BuildMonitorFocusLifecycleTest` covers the service-level startPolling/stopPolling lifecycle.
  - **Reroute scope fix (2026-05-20):** `reroutableFocusPr` now passes `WorkflowContextService.state.value.focusPr?.repoName` to `findMatchingPrForBranch` as `preferredRepoName`. Without this, a cross-repo source-branch-name collision silently rewrote `focusPr` to whichever repo had the higher prId — typically the primary repo — and the dropdown reverted within ~1s after the user selected a non-primary PR on the PR tab. `findMatchingPrForBranch` is now strict on scope: when `preferredRepoName` is set, foreign-repo PRs are excluded even when they have a higher prId; if no PR exists in the focused repo for that branch, returns null and focus is left untouched. Regression scenarios (7a/7b/7c) live in `BuildDashboardPanelDropdownTest`.
- `BuildLogParser` — extracts errors, test failures, CVE warnings from build logs
- `CveRemediationService` — parses CVE data, provides version bump suggestions
- `PlanDetectionService` — auto-detects Bamboo plan key from project
- **Plan/build identity is API-sourced, never inferred from key shape (2026-05-27).** `BambooPlanResolver` (`bamboo/service/`) maps `(masterKey, branchName, repoDefaultBranch?)` → typed `BambooPlanRef` via `getPlanBranches` (`/plan/{key}/branch`, probe-validated 200 on DC 10.2.14). Three states: child `BranchPlan` (its own distinct key, e.g. `PROJ-PLAN138`), `MasterTrackedBranch` (branch == repo default branch → master key is the target), or `null` (no build for this branch → strict empty state, **no master substitution** on the focus path). `BambooApiClient.getLatestResult(planKey)` takes a resolved key and always issues `/result/{key}/latest` — the `/result/{key}/branch/{name}/latest` form is intentionally NOT used (probe: 404s when the branch is the master's tracked branch, and absent from the 10.2.14 swagger). The old `BRANCH_PLAN_KEY_REGEX` (`^.+-.+-\d+$`) and `.last().isDigit()` heuristics are **deleted** — they could not distinguish a master key ending in a digit (`PROJ-BUILD2`) from a branch plan key (`PROJ-PLAN138`); the regex misfired on the real `PROJ-PLAN138` shape and was the cause of the Build-tab stages/jobs regression (commit `def4cda98`). `PlanDetectionService.resolvePlanRef(...)` is the typed entry point; `resolveBranchKey` (waterfall, falls back to master) and `resolveBranchKeyOrNull` (strict null) delegate to the resolver. `BambooBuildStructureMapper.toBuildState(dto, planKey, branch, jobOrder)` is the single stage/job mapper for the live monitor (and the model the service uses). Spec/plan: `docs/superpowers/{specs,plans}/2026-05-27-bamboo-plan-identity*`.
- **Build-tab job order (2026-05-27):** Bamboo's result endpoint returns a stage's jobs in an *unstable* order (the probe returned the same plan's jobs in different orders across builds) with no order field, so the Build tab showed jobs out of order vs the website. The canonical order comes from the plan DEFINITION: `BambooApiClient.getPlanStructure(planKey)` → `GET /plan/{key}?expand=stages.stage.plans.plan` returns `stages.stage[].plans.plan[]` in plan-defined order. `BambooPlanJobOrder.fromConfig` reduces it to `stageName → [job shortName,…]`; `BuildMonitorService` caches it per plan key (cached only on success, so a transient failure just retries next poll) and passes it to `BambooBuildStructureMapper.toBuildState`, which **stable-sorts jobs within each stage** by that order (stages already arrive correctly ordered; matched by job `shortName`). Empty/missing order map or unmatched jobs → original API order preserved. Pinned by `BambooPlanJobOrderTest` + `BambooBuildStructureMapperTest`. Docker-tag detection is already order-independent, so this is display-only.

## Bamboo→Bitbucket bridge (2026-05-07 audit)

- `BambooApiClient.getResultVcsRevision(resultKey)` queries `?expand=vcsRevisions` and returns the first commit SHA recorded against a build. Returns null when Bamboo has no VCS revision recorded for the build (rare — only if the plan has no source repository linked).
- `BuildFailureBridgeStartupActivity` (in `bamboo/listeners/`) wires the bridge: subscribes to `WorkflowEvent.BuildFinished` (FAILED), resolves the commit SHA via `BambooApiClient.getResultVcsRevision`, then calls `:core`'s `BitbucketBranchClient.getCommitPullRequests` for every configured repo and surfaces a single notification (group `workflow.pr`) listing the affected PR ids.
- Lives in `:bamboo` (not `:pullrequest`) because the project's module-graph rule forbids `:pullrequest → :bamboo`. `:bamboo`'s only cross-module dependency stays `:core` — the listener consumes `BitbucketBranchClient` from `:core`, not `:pullrequest`.
- Rich build status (`/rest/api/latest/.../commits/{cid}/builds`) and deployments are deferred — the org's Bamboo agents publish only the basic v1 schema. See `docs/research/2026-05-07-bitbucket-recommendations.md` §2 + §5 for detail.
