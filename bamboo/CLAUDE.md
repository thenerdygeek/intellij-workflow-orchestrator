# :bamboo Module

Build monitoring, CI feedback, CVE remediation. PR display only — creation lives in `:pullrequest`.

## PrBar (Build-tab PR strip)

`PrBar` is a **passive mirror** of the PR tab's selection (redesigned 2026-04-27). It does not fetch PRs from Bitbucket itself, does not poll, and has no Create PR button. It subscribes to `WorkflowContextService.state` and renders:

- **PR-rendered state** (`interactionMode == Live` AND `focusPr != null` — i.e., the focused PR's source branch matches the local checkout): single `PR #id  fromBranch → toBranch` strip with "Open in browser" link.
- **Empty state** otherwise: blue strip with *"No PR for this branch — pick one in the PR tab →"* (clickable, switches to the PR tab).

PR title is intentionally not rendered (`PrRef` doesn't carry it; re-introducing a Bitbucket fetch would defeat the redesign). To restore titles, extend `PrRef.title` from the PR tab side rather than fetching in PrBar.

## Build-details branch-match guard

When `WorkflowContext.interactionMode == ReadOnly` (focused PR's branch ≠ local checkout), the action toolbar's **Refresh**, **Trigger Build**, **Trigger Manual Stage** buttons are disabled with a tooltip identifying the focused PR's branch. The build list, jobs panel, and log viewer stay visible and readable. Implemented via `BuildDashboardActionGate.isLiveMode(ctx)` + a `WorkflowContextService.interactionModeFlow` collector that calls `cachedActionToolbar.updateActionsImmediately()` on transitions.

## PR creation entry point

PR creation only exists on the PR tab now. `:bamboo` no longer depends on `:pullrequest` (delegation happens via the `createPrLauncher` extension point in `:core`); the EP signature is `launch(project, scope)` — the `autoGenerateDescription` parameter was removed in the same redesign.

`PrDescriptionGenerator` and `MarkdownToHtml` live in `:pullrequest/service/`.

## Bamboo Server REST API

Auth: `Authorization: Bearer <PAT>`
Base: `https://{host}/rest/api/latest/`

Key endpoints:
- `GET /rest/api/latest/currentUser` — test connection
- `GET /rest/api/latest/result/{planKey}/latest` — latest build result
- `GET /rest/api/latest/result/{buildKey}` — specific build result + stages
- `GET /rest/api/latest/result/{buildKey}/log` — build log
- `POST /rest/api/latest/queue/{planKey}` — trigger build (with variables)
- `GET /rest/api/latest/result/{planKey}` — running/queued builds

Build variables include `dockerTagsAsJson` — JSON payload of service-to-docker-tag mappings used by automation suites.

## Architecture

- `BambooApiClient` — HTTP client for all Bamboo REST calls
- `BambooServiceImpl` — implements `BambooService` (in :core), returns `ToolResult<T>`
- `BambooServiceImpl` returns `BuildResultData.stages[].jobs[]` populated from `?expand=stages.stage.results.result` responses. Each `BuildJobData.resultKey` (e.g. `PROJ-PLAN138-UNIT-4`) is what the agent passes to `bamboo_builds.get_build_log` for per-job logs.
- `BuildPlanResolutionPolicy` (in `:bamboo/ui/`) is a pure object encapsulating the Build dashboard's "use detected plan / use configured master / show hint" decision, so the resolution matrix is testable without IntelliJ infrastructure. The dashboard delegates to it after every `BambooService.autoDetectPlan` call.
- `BuildMonitorService` — background polling via SmartPoller, emits `BuildFinished`/`BuildLogReady` events. Uses platform-injected `cs: CoroutineScope` (see `:core` "Service & threading conventions"). `PrBar` git reads use `readAction { }`.
- `BuildLogParser` — extracts errors, test failures, CVE warnings from build logs
- `CveRemediationService` — parses CVE data, provides version bump suggestions
- `PlanDetectionService` — auto-detects Bamboo plan key from project

## UI

- `BuildDashboardPanel` — build list + stage detail + log viewer
- `BuildStatusBarWidget` — build status in status bar
- `BuildStatusNodeDecorator` — project tree build status badges
- `CveAnnotator` / `CveIntentionAction` — inline CVE warnings + auto-fix in pom.xml
- `BambooBuildConfigurationType` — run configuration for manual stage triggers
- **UI Overhaul:** StitchLeftAccentBorder utility for status-colored left borders on cards. Uppercase section headers, monospace build numbers in cell renderers.
