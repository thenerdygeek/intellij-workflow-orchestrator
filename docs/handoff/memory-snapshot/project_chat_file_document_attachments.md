**SHIPPED 2026-05-27 (unpushed) on `feature/cross-ide-delegation`: agent chat file & document attachments.**

Root cause of the original bug: the React picker (`<input type=file>`) and drag-drop are correct for a browser but dead in JCEF — no `CefDialogHandler` (picker click does nothing) and no Swing `DropTarget` (OS drops never reach Chromium). Also the pipeline was image-only (png/jpeg/webp). (The "Enable visual support" master switch gating images was a red herring — user already had it on.)

Fix (spec `docs/superpowers/specs/2026-05-27-...-design.md`, plan `docs/superpowers/plans/2026-05-27-...` — note `plans/` is gitignored): acquire files on the JVM (native `FileChooser` via `_pickAttachment` bridge + AWT `DropTarget` on the JCEF component), store bytes in `{sessionDir}/attachments/` (images) or `attachments/files/<sha8>-<name>` (everything else), push only chip metadata over the bridge. Routing: image→vision (existing `ContentBlock.ImageRef` path); any other file→`<attached_files>` path marker appended to the LLM message + auto-activate deferred `read_document` (agent reads on demand with read_file/read_document). File attach is always-on (independent of visual-support); drop overlay driven from JVM `DropTarget` callbacks via `_setDropActive`. New: `AttachmentIngestService`, `AttachmentPicker`, `AttachmentDropTarget`, `AttachmentStore.storeFileBlocking`, `AgentService.activateDeferredTool`, webview `DropOverlay` + `ChipPreview` file chips + `addExternalChip`.

12 task commits `7d8757bab..b2932ec97` + final-review fix `8f12901d1`. Verified: :core/:agent tests, webview vitest 558/558, verifyPlugin (3 IDEs) all green.

Source-text wiring tests can't catch this (they only grep for the symbol).
