## SHIPPED 2026-06-02 — SANITIZER_TIMEOUT root causes (v0.86.0-web-rc8, commit 86565e31e, NOT pushed)

Field report: `web_fetch` returned `SANITIZER_TIMEOUT` on EVERY page incl. minimal example.com. Systematic-debugging → 4 root causes, fixed via TDD (all green: :core/:web/:agent tests + verifyPlugin Compatible). User's hint ("check how the haiku id is extracted") nailed the keystone:

- **⚠ D (KEYSTONE, reusable trap):** `ModelCache.pick{Haiku,SonnetThinking,SonnetNonThinking,Cheapest,Best,buildFallbackChain}` all hard-gated `models.filter { it.provider == "anthropic" }`. `ModelInfo.provider = id.substringBefore("::","unknown")` — so on **Anthropic-Vertex** (id prefix `anthropic-vertex::…`, or ids with NO `::` → provider=`"unknown"`) the gate matched NOTHING → pickHaiku/pickSonnet returned null. Fixed: new private `List<ModelInfo>.anthropicFamily()` matches by model NAME (claude/haiku/sonnet/opus are Anthropic-exclusive; `modelName` falls back to full id) OR provider-contains-anthropic. `HaikuPhraseGenerator` only "worked" because `pickCheapest` has a step-3 "return anything" fallback the others lack. See [[project_anthropic_vertex_assistant_prefill_rejection]] (same Vertex instance).
- **D2:** `LlmBrainFactory.createForSanitization` fell through to `create(project)` = configured heavy Opus-**thinking** model when picks were null → thinking latency + reasoning tokens blew the 60s budget. Added `?: pickCheapest` before that fallback.
- **C:** sanitizer ran `maxOutputTokens=8000` but `webMaxExtractedChars=32768`; it echoes the page VERBATIM, so large pages truncated → incomplete JSON → parse-fail. New `SubagentSpawnerAdapter.outputBudgetForInput(chars)` = chars/3 + 2000, floored 8000, capped 32000.
- **A+B:** `parseResult` returned `Verdict.TIMEOUT` on parse/field-extraction failure (mislabel) → now `UNRECOGNISED` with notes; only the real `withTimeoutOrNull` path yields TIMEOUT. `WebError.SanitizerTimeout` was a no-detail `object` discarding notes → now `class SanitizerTimeout(detail)`; new recoverable `SanitizerUnreadable(detail)` backs the UNRECOGNISED mapping; `WebFetchEngine` surfaces `san.notes`.

⚠ The example.com case is NOT independently confirmed-fixed — D is the strongest hypothesis but needs the user's re-run (the now-surfaced notes will say true-timeout vs parse-fail). rc8 bumped but NOT pushed/released yet (awaiting user).

## SHIPPED 2026-06-03 — web_fetch made functional (v0.86.0-web-rc9, commit aabd5d100, pushed + released)

rc8's model-resolution fix EXPOSED the next layer (classic). Field test of rc8 → 3 bugs, TDD-fixed (all green: :core/:web/:agent 4677 + verifyPlugin Compatible):

- **⚠ Bug1 (CRITICAL, architecture):** `SANITIZER_UNREADABLE` note = **"Exceeded maximum iterations (1)"** on EVERY page incl. example.com. The sanitizer is **single-shot text-in/text-out** (persona returns raw JSON, never calls `task_report`), but `SubagentSpawnerAdapter` ran it through the agentic `SubagentRunner`/`AgentLoop` — which only emits `LoopResult.Completed` on a terminal tool call. Rewrote runSanitizer/Batch to a single `brain.chat()` completion (no loop/tools/iteration cap), mirroring `HaikuPhraseGenerator`; added injectable `brainFactory` seam for tests. LESSON: don't run a single-shot text generator through the ReAct loop; the loop needs `task_report`. If PromptExtractor/egress screener also use SubagentRunner, check them.
