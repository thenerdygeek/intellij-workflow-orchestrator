# Aider, SWE-agent, Codex CLI: Sub-agent and Delegation Patterns

Source-code-level research from repos: paul-gauthier/aider, SWE-agent/SWE-agent, openai/codex (March 2026)

## 1. Aider: Architect/Editor Two-LLM Delegation

### Architecture: NOT sub-agents. Two-phase synchronous pipeline.

Aider has NO sub-agent system. Instead, it uses a **two-LLM pipeline** where an "architect" LLM plans changes and a cheaper "editor" LLM executes them.

### Class Hierarchy

```
Coder (base_coder.py)
  -> AskCoder (ask_coder.py) — just answers questions, no edits
    -> ArchitectCoder (architect_coder.py) — plans changes, delegates to editor
  -> EditBlockCoder (editblock_coder.py) — applies SEARCH/REPLACE blocks
    -> EditorEditBlockCoder (editor_editblock_coder.py) — editor variant
  -> WholeFileCoder, PatchCoder, UnifiedDiffCoder — other edit formats
```

### Exact Architect-to-Editor Delegation Flow

From `architect_coder.py` (complete file, 49 lines):

```python
class ArchitectCoder(AskCoder):
    edit_format = "architect"
    gpt_prompts = ArchitectPrompts()
    auto_accept_architect = False

    def reply_completed(self):
        content = self.partial_response_content  # Architect LLM output

        if not content or not content.strip():
            return

        if not self.auto_accept_architect and not self.io.confirm_ask("Edit the files?"):
            return

        kwargs = dict()

        # Use the editor_model from the main_model if it exists, otherwise use the main_model itself
        editor_model = self.main_model.editor_model or self.main_model

        kwargs["main_model"] = editor_model
        kwargs["edit_format"] = self.main_model.editor_edit_format
        kwargs["suggest_shell_commands"] = False
        kwargs["map_tokens"] = 0
        kwargs["total_cost"] = self.total_cost
        kwargs["cache_prompts"] = False
        kwargs["num_cache_warming_pings"] = 0
        kwargs["summarize_from_coder"] = False

        new_kwargs = dict(io=self.io, from_coder=self)
        new_kwargs.update(kwargs)

        editor_coder = Coder.create(**new_kwargs)
        editor_coder.cur_messages = []
        editor_coder.done_messages = []

        editor_coder.run(with_message=content, preproc=False)

        self.move_back_cur_messages("I made those changes to the files.")
        self.total_cost = editor_coder.total_cost
        self.aider_commit_hashes = editor_coder.aider_commit_hashes
```

### Key Design Decisions

1. **Architect prompt** (exact system prompt):
   ```
   Act as an expert architect engineer and provide direction to your editor engineer.
   Study the change request and the current code.
   Describe how to modify the code to complete the request.
   The editor engineer will rely solely on your instructions, so make them unambiguous and complete.
   Explain all needed code changes clearly and completely, but concisely.
   Just show the changes needed.
   DO NOT show the entire updated function/file/etc!
   ```

2. **Editor prompt** (exact system prompt):
   ```
   Act as an expert software developer who edits source code.
   Describe each change with a *SEARCH/REPLACE block* per the examples below.
   All changes to files must use this *SEARCH/REPLACE block* format.
   ONLY EVER RETURN CODE IN A *SEARCH/REPLACE BLOCK*!
   ```

3. **Context sharing via `from_coder`**: The editor inherits `abs_fnames`, `abs_read_only_fnames`, `done_messages`, `cur_messages`, `commands`, `file_watcher` from the architect. But `summarize_from_coder=False` means it does NOT summarize history.

4. **The editor gets zero repo map** (`map_tokens=0`), no shell commands, no cache warming. It's a pure "apply these edits" call.

5. **Editor model defaults**: The model can have a separate `editor_model_name` and `editor_edit_format` configured. If edit format is "diff"/"whole"/"diff-fenced", it's auto-prefixed with "editor-". Default editor format is "editor-diff" (SEARCH/REPLACE blocks).

6. **Cost tracking flows back**: `self.total_cost = editor_coder.total_cost` and `self.aider_commit_hashes = editor_coder.aider_commit_hashes`.

7. **No multi-turn editor**: The editor runs ONCE with `with_message=content` and returns. No iteration.

### How Aider Decides Search vs Act

Aider does NOT have the LLM decide this. The **edit format** is fixed at startup (chosen by CLI flags or model config). The architect always plans; the editor always edits. There is no tool-use decision. The repo map (generated by tree-sitter) gives the LLM structural context so it knows what exists.

## 2. SWE-agent: Single ReAct Loop + Retry Outer Loop

### Architecture: Single agent loop, NO sub-agents. Optional retry-with-review outer loop.

SWE-agent has two agent types but they are NOT sub-agents in the traditional sense:

```python
AgentConfig = DefaultAgentConfig | RetryAgentConfig | ShellAgentConfig
```

### DefaultAgent: The Core Single Loop

```python
class DefaultAgent:
    def run(self, env, problem_statement, output_dir):
        self.setup(env=env, problem_statement=problem_statement, output_dir=output_dir)
        self._chook.on_run_start()
        step_output = StepOutput()
        while not step_output.done:
            step_output = self.step()
            self.save_trajectory()
        # ...

    def step(self):
        step_output = self.forward_with_handling(self.messages)
        self.add_step_to_history(step_output)
        self.add_step_to_trajectory(step_output)
        return step_output
```

The `forward_with_handling` method queries the LLM, parses the action, executes it in the environment, and handles errors (format errors, blocked actions, bash syntax errors) with up to `max_requeries=3` retries.

### RetryAgent: Outer Loop with Review

`RetryAgent` wraps `DefaultAgent` for **multiple full attempts** at solving a problem:

```python
class RetryAgent:
    def __init__(self, config: RetryAgentConfig):
        self._agent: DefaultAgent | None = None  # Current attempt's agent
        self.config.agent_configs  # Can have DIFFERENT configs per attempt

    def run(self, env, problem_statement, output_dir):
        self._setup_agent()  # Creates a DefaultAgent
        while not step_output.done:
            step_output = self.step()  # Delegates to self._agent.step()
            if step_output.done:
                self._rloop.on_submit(ReviewSubmission(...))  # Review the attempt
                self._finalize_agent_run()
                if self._rloop.retry():  # Should we try again?
                    self._next_attempt()  # env.hard_reset() + new DefaultAgent
                    step_output.done = False
```

### Two Review Strategies

1. **ScoreRetryLoop**: A separate reviewer LLM scores each attempt (1-10). Accepts if score >= threshold. Picks best score.
   ```python
   class Reviewer:
       def review(self, instance, submission):
           messages = self.format_messages(instance, submission)
           for _ in range(self._config.n_sample):  # Multiple samples for robustness
               answer = self._model.query(messages)
               score = self.interpret(answer)  # Extract numeric score
           return ReviewerResult(accept=avg_score - penalty)
   ```

2. **ChooserRetryLoop**: Run N attempts, then a "chooser" LLM picks the best one:
   ```python
   class Chooser:
       def choose(self, problem_statement, submissions):
           # Optional preselector filters to top candidates
           messages = self.build_messages(problem_statement, filtered_submissions)
           response = self.model.query(messages)
           chosen_idx = self.interpret(response)  # Extract index number
   ```

### ACI (Agent-Computer Interface) Design

SWE-agent's ACI is defined through **tool bundles** -- YAML config files with custom bash commands:

```yaml
# tools/search/config.yaml
tools:
  find_file:
    signature: "find_file <file_name> [<dir>]"
    docstring: "finds all files with the given name or pattern in dir"
    arguments:
      - name: file_name
        type: string
        required: true
      - name: dir
        type: string
        required: false
  search_dir:
    signature: "search_dir <search_term> [<dir>]"
    docstring: "searches for search_term in all files in dir"
  search_file:
    signature: "search_file <search_term> [<file>]"
    docstring: "searches for search_term in file"
```

Key ACI design principles:
- **Custom commands replace raw bash for common operations** -- `find_file`, `search_dir`, `search_file`, `open`, `goto`, `scroll_down/up`, `edit`, `create`
- **Tool documentation is injected into system prompt** via `{{command_docs}}` template variable
- **Blocklist prevents interactive commands**: vim, vi, emacs, nano, less, `tail -f`, `python -m venv`, make, standalone python/bash/sh
- **Observation truncation**: Max 100K chars, with template telling agent to use head/tail/grep
- **State command**: After each action, a "state command" runs to show current context (e.g., current file, line numbers)
- **Function calling parsing**: Modern SWE-agent uses native function calling (`type: function_calling`) rather than text parsing
- **Review on submit**: The `review_on_submit_m` bundle shows the diff and asks the agent to verify before final submission

### How SWE-agent Guides Tool Selection

From `config/default.yaml`, the instance template includes explicit step-by-step guidance:
```
1. As a first step, it might be a good idea to find and read code relevant to the <pr_description>
2. Create a script to reproduce the error and execute it with `python <filename.py>` using the bash tool
3. Edit the sourcecode of the repo to resolve the issue
4. Rerun your reproduce script and confirm that the error is fixed!
5. Think about edgecases and make sure your fix handles them as well
```

This is the ONLY mechanism for guiding search-vs-act decisions. There is NO dynamic tool selection, no routing, no planner. The LLM decides based on prompt guidance + demonstration trajectories.

## 3. OpenAI Codex CLI: Full Multi-Agent with Thread Spawning

### Architecture: TRUE sub-agents via thread spawning. Parent-child hierarchy.

Codex CLI has the most sophisticated sub-agent system of the three. It's built in Rust with:

### Core Components

1. **AgentControl** (`agent/control.rs`): Control plane for multi-agent operations. One per root session, shared with all sub-agents.
   ```rust
   pub(crate) struct AgentControl {
       manager: Weak<ThreadManagerState>,  // Weak to avoid reference cycles
       state: Arc<AgentRegistry>,
   }
   ```

2. **AgentRegistry** (`agent/registry.rs`): Tracks spawned agents, enforces limits.
   ```rust
   pub(crate) struct AgentRegistry {
       active_agents: Mutex<ActiveAgents>,
       total_count: AtomicUsize,
   }
   ```

3. **Five tool handlers** for agent management:
   - `spawn_agent` -- Create a new sub-agent thread
   - `wait_agent` -- Block until sub-agent(s) complete
   - `send_input` -- Send messages to a running sub-agent
   - `close_agent` -- Terminate a sub-agent
   - `resume_agent` -- Resume a paused agent

### spawn_agent Tool (What the LLM Sees)

The tool description includes EXPLICIT guidance for when to delegate:

```
Only use `spawn_agent` if and only if the user explicitly asks for sub-agents,
delegation, or parallel agent work. Requests for depth, thoroughness, research,
investigation, or detailed codebase analysis do not count as permission to spawn.

Spawn a sub-agent for a well-scoped task. This spawn_agent tool provides you
access to smaller but more efficient sub-agents.

### When to delegate vs. do the subtask yourself
- Use the smaller subagent when a subtask is easy enough for it to handle and can
  run in parallel with your local work.
- Prefer delegating concrete, bounded sidecar tasks that materially advance the main
  task without blocking your immediate next local step.
- Delegate verification only when it can run in parallel with ongoing implementation
  and is likely to catch a concrete risk before final integration.
- The key is to find opportunities to spawn multiple independent subtasks in parallel
  within the same round, while ensuring each subtask is well-defined, self-contained,
  and materially advances the main task.
```

### spawn_agent Parameters

```rust
struct SpawnAgentArgs {
    message: Option<String>,        // Initial task description
    items: Option<Vec<UserInput>>,  // Rich input items (alternative to message)
    agent_type: Option<String>,     // Role name (e.g., "researcher", "coder")
    model: Option<String>,          // Override model for sub-agent
    reasoning_effort: Option<ReasoningEffort>,  // Control reasoning depth
    fork_context: bool,             // Fork parent's conversation history
}
```

### Role System

Roles are configuration layers applied at spawn time (`agent/role.rs`):
- Built-in roles: "default", plus user-defined roles via config files
- Each role can have its own system prompt, model, tools, config overrides
- Role config is a TOML file layered on top of parent config
- Parent's model/provider preserved unless role explicitly overrides

### Depth Limiting

```rust
pub(crate) fn exceeds_thread_spawn_depth_limit(depth: i32, max_depth: i32) -> bool {
    depth > max_depth
}
// Error message: "Agent depth limit reached. Solve the task yourself."
```

### Fork Mode

When `fork_context=true`, the parent's full conversation history is copied to the child thread:
```rust
const FORKED_SPAWN_AGENT_OUTPUT_MESSAGE: &str =
    "You are the newly spawned agent. The prior conversation history was forked from
     your parent agent. Treat the next user message as your new task, and use the
     forked history only as background context.";
```

### Wait Mechanism

The parent agent calls `wait_agent` with target thread IDs and a timeout:
```rust
struct WaitArgs {
    targets: Vec<String>,
    timeout_ms: Option<i64>,  // Default: some constant, clamped to min/max
}
```
Returns `HashMap<String, AgentStatus>` with each agent's final status. Uses tokio watch channels for async notification.

### SubAgentSource Types

```rust
enum SubAgentSource {
    Review,              // Internal review sub-agent
    Compact,             // Context compaction sub-agent
    MemoryConsolidation, // Memory management sub-agent
    ThreadSpawn { parent_thread_id, depth, agent_path, agent_role, .. },  // User-visible spawn
    Other(String),       // Generic labeled sub-agent
}
```

## 4. Comparative Analysis: Search vs. Act Decision Patterns

### Aider
- **No dynamic decision**: Edit format is fixed. Architect always plans, editor always edits.
- **Repo map** (tree-sitter) gives structural context so the LLM implicitly knows what to search.
- **User controls**: `/add` files to context, `/architect` or `/code` mode switch.

### SWE-agent
- **Prompt-guided only**: Step-by-step instructions in instance template ("first find relevant code, then reproduce, then edit, then verify").
- **Demonstrations**: Pre-recorded trajectories shown as few-shot examples.
- **No router/classifier**: Single LLM decides everything based on prompt guidance.
- **State visibility**: After each command, state command shows current file/position, giving the LLM spatial awareness.

### Codex CLI
- **Tool description guidance**: The `spawn_agent` description explicitly says when to delegate vs. do locally.
- **Model-based routing**: Sub-agents can use different/cheaper models than the parent.
- **Parallel execution**: Multiple sub-agents can run simultaneously; parent waits for all.
- **Role-based specialization**: Different system prompts and tools per agent type.
- **Explicit user permission required**: "Only use spawn_agent if the user explicitly asks for sub-agents."

### Common Patterns Across All Three

1. **None use a separate "router" or "planner" agent** to decide what to do. The main LLM always decides.
2. **All use prompt engineering** rather than learned policies for action selection.
3. **Cost tracking always flows back** to the parent/caller.
4. **All limit context for delegated work**: Aider zeros out repo map for editor; SWE-agent truncates observations; Codex can fork or start fresh.
5. **Error recovery through re-query**: All three retry on format/syntax errors before giving up.

## 5. Key Architectural Insights for Plugin Design

1. **Aider's architect/editor is the simplest delegation pattern**: One LLM plans in natural language, another executes as SEARCH/REPLACE. No tool calls needed for the delegation itself.

2. **SWE-agent proves single-loop is competitive**: Despite having no sub-agents, SWE-agent achieves strong benchmark results. The ACI (custom commands, state visibility, observation truncation) matters more than multi-agent.

3. **Codex CLI's spawn model is the most production-ready**: Thread-based, with proper lifecycle (spawn/wait/send/close/resume), depth limits, cost budgets, role configs. But notably requires EXPLICIT user permission to spawn.

4. **For our Cody-based agent**: The Aider architect/editor pattern is most applicable -- we can use Cody as the "architect" (plans changes using context from our tools), then apply edits programmatically (no need for an "editor LLM"). This avoids the complexity of thread management while getting the key benefit of separating planning from execution.
