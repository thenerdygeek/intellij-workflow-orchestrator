**Goal:** Split Workflow Orchestrator into two installable IntelliJ plugins: open-source **A** (standalone-useful AI agent + de-conventioned Atlassian/Sonar/Sourcegraph connectors) and private **B** that `<depends>` on A via extension points (NEVER a code fork — B plugs into A's sockets and overrides via the existing EP+default+runCatching pattern) and welds the company's workflow on top.

**Decisions (2026-06-22 brainstorm):**
- Identity: open AI coding agent for **IntelliJ IDEA, Java/Kotlin/Python**, Atlassian-stack reference connectors. NOT a universal JetBrains agent (hard `com.intellij.modules.java` dep at `plugin.xml:9` bounds it to IDEA; built against Ultimate).
- Scope = **Tier 1**: strip company *conventions*, keep Atlassian connectors in A behind neutrally-named seams; do NOT build GitHub/GitLab/Jenkins/Anthropic impls (Tier 2 = deferred/community). Seams are named+shaped now so Tier 2 is additive.
- Jira = **Server/DC only** (Cloud = future adapter: different auth/accountId/ADF). LLM seam is DEEP — XML-in-content tool calling is a Cody workaround; a native-function-calling provider = Tier 2.
- B holds: `:automation` (DockerTagsAsJSON, "Unique Docker Tag" log regex, semver baseline scoring), `:handover` (minus CopyrightFix year-logic→A), Bamboo→Bitbucket failure bridge, conventions/default values, copyright template, devops persona, proprietary personas.
- A/B test: generic mechanism → A (overridable policy); company-only tooling → B. Caught: AI PR review is GENERIC (methodology lives in swappable persona) → A.
- Security: plugin SOURCE is clean (placeholders/RFC-5737 only); the REPO (docs/audit/research/CLAUDE.md/fixtures) has real-infra refs ("DC 10.2.14", real plan keys) → A's public repo must start FRESH + gitleaks gate before publish.

**Rev-2 decisions (2026-06-22):**
- **SEQUENCING = internal-first (open-source LATER).** Build the two-plugin split + de-convention + LLM seam, ship A+B PRIVATELY, keep A's API `internal`/unfrozen. Freezing `@StableApi` + Marketplace publish = a DEFERRED separate effort (Phase 5), done only once the native provider is proven and A isn't visibly Cody-shaped. Phases 0–4 are private.
- **Configurable-engine framing (3 tiers):** A's A/B test = "can it be config?" Tier1 plain settings (provider choice, URLs, branch pattern, status names) → configure A; Tier2 pluggable impl via config/EP (which LlmProvider/WorkflowConfig) → EP, NOT a fork; Tier3 genuinely new code → B only. So B = OUR config PRESET + tier-3 modules (automation/handover); most adopters never write code.
- **LLM pluggability = FIRST-CLASS workstream** (not Tier 2). Two-layer seam: `LlmProvider` (transport/auth/model-id) + `ToolProtocol` (XML-in-content for Sourcegraph vs NATIVE function-calling for Anthropic). Loop calls ToolProtocol, not hardcoded XML. KEY: native = ONE impl for Anthropic-direct/Bedrock/Vertex (same Messages API tool-use; differ only in Java-SDK backend + model-id: bare `claude-opus-4-8` direct/Vertex, `anthropic.`-prefix Bedrock). NativeProtocol DELETES the DialectDriftDetector machinery.

**Round-2 review (3 lenses: consistency/newcontent-soundness/skeptic) — 2 real blockers fixed in REV 3:**
