**UPDATE 2026-06-29 — runIde NOW WORKS ON THIS MAC.** The Ultimate license wall was resolved (trial activated), so `./gradlew :runIde` (A-alone) and `:plugin-b:runIde` (A+B) both launch, open a project, and fire `[PluginSplit]` markers on THIS Mac — the old "deferred to Windows" blocker is lifted for headless+interactive runIde. cowork drove Phase-2 Part-1 (A-alone, all PASS) and Part-2 live on the Mac sandbox. Two operational traps discovered this session:
- **Sandbox log path (v2):** the IntelliJ Platform Gradle Plugin 2.x sets `-Didea.log.path=<sandbox>/IU-2025.1.7/log`, so idea.log is at `…/IU-2025.1.7/log/idea.log` — **NOT** `…/system/log/idea.log`. A readiness-watcher grepping `system/log` silently never matches (false timeout while the IDE is actually fine). Confirm the path from the `-Didea.log.path` JVM arg.
- **Auto-reload corrupts the A+B sandbox UI.** runIde defaults `-Didea.auto.reload.plugins=true`. The full A+B plugin (companyb especially) is NOT dynamically unload-safe, so when ANY background `./gradlew` rewrites the bundled `automation.jar`/`handover.jar` under a RUNNING sandbox, IntelliJ's failed hot-reload SILENTLY STRIPS Company B's tabs + settings pages (`…companyb.plugin is not unload-safe because class loader cannot be unloaded`) while core `CompanyBWorkflowConfig` survives — looks exactly like a carve regression but is NOT. Operational rule during QA: **never run a gradle build while a QA sandbox is open.** Config fix added (`build.gradle.kts` RunIde block: `-Didea.auto.reload.plugins=false`) BUT `org.gradle.configuration-cache=true` reuses the cached task graph, so a build-script edit to runIde args needs `--no-configuration-cache` (or set the task's own `autoReload` property) to actually take effect — verify the live value in the AppStarter "JVM options" log line, not just the source.
