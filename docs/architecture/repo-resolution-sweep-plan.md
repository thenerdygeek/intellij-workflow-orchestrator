# Repo-resolution sweep plan

Branch: `refactor/cleanup-perf-caching` | Started: 2026-04-27

## Why

`RepoContextResolver.resolveCurrentEditorRepoOrPrimary()` (and its `resolveFromCurrentEditor()` / `getPrimary()` siblings) is the easy-to-call API that always returns a repo. It became the default for any code that needed "a repo" — even when the calling context already had a stronger user-action signal (checked changes, focused PR, selected build, clicked ticket).

Surfaced bug: in a multi-module project (parent has no `.git`, submodules each have their own `.git` — same shape as commit `4be45839`), clicking **Generate with Workflow** in the commit dialog asked git about the editor's submodule, not the submodule the user had checked. Files were filtered out as "outside repoRoot" and the action aborted.

Audit found 14 callers of the editor/primary resolvers; **11** were misuse. This plan fixes them in one sweep.

## Foundation: new helper

`core/settings/RepoContextResolver.kt` gets:

```kotlin
/**
 * Resolves a [GitRepository] from a file system path. Prefers the deepest matching
 * repository root (so a file in a nested submodule resolves to the submodule, not
 * the parent). Returns null if no repository contains the path.
 */
fun findRepositoryForPath(path: String): GitRepository? {
    val repos = GitRepositoryManager.getInstance(project).repositories
    if (repos.isEmpty()) return null
    if (repos.size == 1) return repos.first()
    return repos
        .filter { path == it.root.path || path.startsWith(it.root.path + "/") }
        .maxByOrNull { it.root.path.length }
}
```

The existing private `findRepositoryForFile(file: VirtualFile): GitRepository?` is generalised to delegate to this helper.
