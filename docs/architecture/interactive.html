Agent Architecture (Phase 3)

The :agent module adds an AI coding assistant powered by Sourcegraph Enterprise's LLM API. It features a ReAct loop with 52 tools, LLM-controlled worker delegation, interactive planning, cross-session memory, and user-extensible skills. The agent runs entirely within the IDE, using PSI and IntelliJ APIs for code intelligence.

ReAct Loop & Budget Enforcement

Each user message enters a ReAct (Reason + Act) loop. The LLM reasons about the task, selects tools, observes results, and iterates until done. A BudgetEnforcer monitors token usage across four thresholds to prevent runaway sessions.

Tool Categories (52 tools)

Tools are organized into four categories. Core tools are always available. IDE Intelligence tools leverage IntelliJ's PSI and inspection APIs. Enterprise Integration tools connect to Jira, Bamboo, SonarQube, and Bitbucket. Planning & Memory tools manage plans, questions, memory, and skills.

Three-Layer Plan Persistence

Plans survive context compression through a three-layer persistence strategy. The disk layer (plan.json) provides crash recovery. The context anchor layer keeps the plan visible to the LLM even after compression. The UI layer shows real-time progress in the chat panel and a dedicated editor tab.

Skill System

Skills are user-extensible behavior presets defined in SKILL.md files. They can be activated via slash commands (/skill-name), toolbar dropdown, or by the LLM calling activate_skill. When active, a skill injects a compression-proof anchor into the context and applies preferred tool filtering.

Activation Sources

  • /skill-name slash command in chat
  • Toolbar dropdown selection
  • LLM calls activate_skill tool

What a Skill Provides

  • System prompt instructions (compression-proof anchor)
  • Preferred tool list (subset filtering)
  • Banner in chat UI showing active skill

Worker Delegation

The delegate_task tool spawns a WorkerSession with fresh context, a scoped tool subset, and its own budget. Workers cannot delegate further (single-depth). Results are summarized and returned to the parent session. This enables parallel subtask execution for large refactoring or multi-file operations.
