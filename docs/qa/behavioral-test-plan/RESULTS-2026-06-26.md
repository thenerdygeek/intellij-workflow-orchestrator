## PART 2 — UPDATE: integration BLOCKED by plugin SSRF guard (root cause confirmed)

After the mock was started, re-testing showed Connections for the mock **never persist**. Root cause is confirmed from source + log (full write-up in `docs/qa/MOCK-SERVER-REQUESTS.md` REQ-1):

- **[BUG-CFG-2 / BLOCKER] SSRF guard rejects localhost.** `BaseUrlValidator` (core/security) blocks `localhost`, `127.x`, `::1`, link-local and all RFC-1918 private addresses, and DNS-resolves hostnames too. Log: `idea.log` → `[Settings:Connections] Jira URL rejected by SSRF guard: Loopback address 'localhost' is not allowed (SSRF risk)` (7+ occurrences). Severity **P0 for QA** (cannot point plugin at a localhost mock); by-design for production security.
- **[BUG-CFG-1] apply() aborts all credential saves on one bad URL.** `ConnectionsConfigurable.apply()` does `return` on the first `Invalid` URL — before saving `pendingTokens` and before `pendingTokens.clear()`. Effect: tokens for the mock never persist, the panel is stuck "modified" (Apply never greys), and the only signal is a transient notification + log WARN (no blocking dialog). Severity **P1** (silent credential-save drop + perpetual-dirty state). This is a real defect independent of the mock.
- Earlier-noted CFG-OBS-2 ("Configure Sourcegraph token first" despite a token in the field) is explained by this: the token was never persisted because apply aborted.
- Connectivity proof: Sourcegraph **Test Connection** → "Connection refused on :8088" earlier (mock down) and the config never saved (SSRF abort) — so the loopback URL also can't even be stored.

### Priority-1 verdict: 🚫 BLOCKED (cannot configure plugin → mock in this build)
The Agent integration gap is **NOT closed** — not because the wire format is wrong (untested), but because the plugin's SSRF guard refuses every localhost/private URL and `apply()` drops the credentials. Unblock requires either a plugin dev-bypass + rebuild, or exposing the mock on a public hostname via a tunnel (see MOCK-SERVER-REQUESTS REQ-1, options 1-3). The high-risk XML-tool-call dialect question remains unproven until configuration is possible.
