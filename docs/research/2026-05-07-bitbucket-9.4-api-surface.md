## 2. New / missed endpoints in DC 9.4

### 2.1 Builds & Deployments (NEW first-party REST API)

This is the single biggest delta vs the v0 probe. Atlassian moved build status into core REST starting **DC 7.4** (`builds`) and added the deployment-status companion in **DC 7.16** (`deployments`). Both live under `/rest/api/latest/...`, NOT under `/rest/build-status/`. The legacy `/rest/build-status/1.0/` is still served for back-compat but the "rich" features (test counts, parent SHA, required-builds enforcement) only flow through the new path.

Sources: [v904 builds-and-deployments](https://developer.atlassian.com/server/bitbucket/rest/v904/api-group-builds-and-deployments/), [updating-build-status how-to](https://developer.atlassian.com/server/bitbucket/how-tos/updating-build-status-for-commits/), [Jenkins plugin deployment_notifications.md](https://github.com/jenkinsci/atlassian-bitbucket-server-integration-plugin/blob/master/docs/deployment_notifications.md).

| Method | Path | R/W | Note |
|---|---|---|---|
| GET | `/rest/api/latest/projects/{p}/repos/{r}/commits/{cid}/builds` | R | **Replaces probe `build_status_v1`/`build_status_v2`.** Returns full schema incl. `key`, `state`, `url`, `buildNumber`, `description`, `duration`, `lastUpdated`, `name`, `parent`, `ref`, `testResults: { failed, skipped, successful }`. |
| POST | `…/commits/{cid}/builds` | W (mutating) | Submit build status. **Add to `_WRITES_INVENTORY`.** |
| DELETE | `…/commits/{cid}/builds?key={buildKey}` | W (mutating) | Delete one build status by key. **Add to `_WRITES_INVENTORY`.** |
| GET | `/rest/api/latest/projects/{p}/repos/{r}/commits/{cid}/deployments` | R | List deployments associated with this commit. NEW SURFACE — completely missing from v0. |
| POST | `…/commits/{cid}/deployments` | W (mutating, REPO_READ) | Register deployment. Body: `key`, `state ∈ {SUCCESSFUL,FAILED,IN_PROGRESS,CANCELLED,UNKNOWN}`, `displayName`, `url`, `environment: { key, displayName, type }`, `deploymentSequenceNumber`. **Add to `_WRITES_INVENTORY`.** |
| DELETE | `…/commits/{cid}/deployments?key=…&environmentKey=…&deploymentSequenceNumber=…` | W (mutating, REPO_ADMIN) | Delete deployment. **Add to `_WRITES_INVENTORY`.** |
| GET | `/rest/api/latest/build/capabilities` | R | Discovery: what build-status fields this instance supports. Already inventoried — should be **probed**. |
| GET | `/rest/api/latest/deployment/capabilities` | R | Discovery: deployment-status capability advertise. NEW. **Probe this.** |

### Build-status v1 vs v2 schema delta

There is **no `/rest/build-status/2.0/`**. Atlassian's "v2" lives at `/rest/api/latest/.../commits/{cid}/builds`. The schema delta vs `/rest/build-status/1.0/` is:

| Field | v1 (`/rest/build-status/1.0/`) | "v2" (`/rest/api/latest/.../builds`) |
|---|---|---|
| `key` | yes | yes |
| `state` | `SUCCESSFUL\|FAILED\|INPROGRESS` | `SUCCESSFUL\|FAILED\|INPROGRESS\|CANCELLED\|UNKNOWN` |
| `name`, `url`, `description` | yes | yes |
| `dateAdded` | yes | (replaced by) `lastUpdated` |
| `parent` (parent SHA for build pipelines) | **no** | **yes** |
| `ref` (which branch built) | **no** | **yes** |
| `buildNumber` | optional | yes |
| `duration` (ms) | **no** | **yes** |
| `testResults: {successful, failed, skipped}` | **no** | **yes** |

The "rich" fields are exactly what the user's Bamboo↔Bitbucket bridge wants (PR badge "tests: 142/144 passed").

### Endpoints to **remove** from the writes table (deprecated/removed in 9.x)

- **`POST /rest/api/1.0/tasks`** — REMOVED in 9.0 per [API changelog](https://developer.atlassian.com/server/bitbucket/reference/api-changelog/). Not in current `_WRITES_INVENTORY` so nothing to remove, but call out in the table footer that the plugin must not migrate to it.
- **`POST /rest/build-status/1.0/commits/{cid}`** — Soft-deprecated. Not in current `_WRITES_INVENTORY` (plugin doesn't write build status). Should be flagged as "do NOT add" rather than removed.
- **`GET /rest/mirroring/latest/mirrorServers/{mirrorId}/token`** — REMOVED in 9.x for security. Not in inventory but worth noting in the doc footer.

### 6.3 Capabilities-discovery follow-up

The probe currently hard-codes `/rest/insights/1.0/...`. In v1, the order should be:

1. Call `/rest/capabilities` first (already does).
2. Call `/rest/insights/latest/capabilities` second (NEW).
3. Read the `url` field in that response (`{ "url": "/rest/insights/1.0" }` typically).
4. Use that resolved base path for subsequent insights calls.

Same pattern for build (`/rest/api/latest/build/capabilities`) and deployment (`/rest/api/latest/deployment/capabilities`) — although those happen to live under `/rest/api/latest/` directly so the discovery is mostly informational.

The CLI sketch:

```python
def _resolve_insights_base(self) -> str:
    body = _read_raw_body(self.raw_dir / "insights_capabilities.json")
    if isinstance(body, dict) and isinstance(body.get("url"), str):
        return body["url"].rstrip("/")
    return "/rest/insights/1.0"  # fallback
```

Then `f"{self._resolve_insights_base()}/projects/{pk}/repos/{rs}/commits/{cid}/reports"` in subsequent probes.

### 6.4 Server-vs-DC detection heuristic fix

Today the probe relies on `GET /rest/api/1.0/admin/license` returning `dataCenter: true`. That endpoint is **admin-only** — non-admin PATs 401/403 there, leaving the probe with `(unknown — admin-only endpoint)`.

Better: **check if the `repository-mirroring` capability is advertised** in the already-public `/rest/capabilities` response. Smart-mirroring is DC-exclusive (Server SKU can't run it). So:

```python
def _detect_edition(self) -> str:
    body = _read_raw_body(self.raw_dir / "capabilities.json")
    if isinstance(body, dict):
        caps = body.get("capabilities") or body.get("application", {}).get("capabilities") or {}
        keys = caps.keys() if isinstance(caps, dict) else (caps if isinstance(caps, list) else [])
        if any("repository-mirroring" in k or "mirroring" in k for k in keys):
            return "Data Center"
    # Fall back to admin/license if we have it
    ldata = _read_raw_body(self.raw_dir / "admin_license.json")
    if isinstance(ldata, dict) and ldata.get("dataCenter") is True:
        return "Data Center"
    return "unknown"
```

This works on **non-admin tokens** because `/rest/capabilities` is unauthenticated/anonymous-readable. Bonus: also cleanly handles the "Server SKU is dead after 8.x EOL" reality — we no longer need to distinguish, just need to confirm DC.
