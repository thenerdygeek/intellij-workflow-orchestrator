## 1. Version detection — what actually works on 3.90 with a non-admin token

**TL;DR — the existing probe v0 fallback chain is wrong, and the right answer is dirt-cheap and works on every response.**

### 1.1 Phantom — drop in v0.1
- `GET /service/rest/v1/system/about` → **404 on 3.90.** This path was never in the `/service/rest/v1/` namespace; the closest legitimate ancestor is `/service/rest/atlas/system-information` which is admin-gated. Drop from probe v0.1. [verified: probe v0; help-portal listing under api-reference.html does not include `/system/about`]

### 1.2 Recommended primary — the `Server:` HTTP header (works for ANY response)

Every Nexus 3 HTTP response (Nexus REST, web UI, **and** the Docker registry on most deployments) sets:

```
Server: Nexus/3.90.1-01 (PRO)
```

[source: https://support.sonatype.com/hc/en-us/articles/24737888833299 — "HTTP responses from Nexus Repository include a 'Server' header that contains the full product version… 'Server: Nexus/3.63.0-01 (PRO)'"]

Properties:
- **No auth required** to read it (it's set even on 401/403 responses, so it works *before* auth succeeds).
- Returns the **build number** (`-01`) and **edition** (`(PRO)` vs OSS) — both of which the rapture-cache-buster scrape doesn't give you.
- Works on the Docker registry host too — answers "is this Nexus or some other registry?" in one call.
- Survives reverse proxies that don't strip `Server:` (most don't; some hardening configs do — see §5 risk).

**Probe action:** capture `response.headers["Server"]` on every raw/*.json save. Parse `Nexus/(\d+\.\d+\.\d+)-(\d+)\s*\(([^)]+)\)` out of it for version + build + edition. This becomes the single primary version-detect signal.

### 1.3 Recommended secondary — `GET /service/rest/v1/status/check`

[source: https://help.sonatype.com/en/sonatype-nexus-repository-3-81-0-release-notes.html — "/service/metrics/healthcheck → /service/rest/v1/status/check" (renamed in 3.81)]

- **Auth: BasicAuth required**, but no admin privilege documented; returns `200` for any authenticated user with read permissions.
- Returns `map[string]Result` — full system-check JSON. Body is informational; the raw response and HTTP status code together already tell us "Nexus 3.81+ AND token has at least read perms".
- Combined with §1.2, this gives "version" PLUS "is the v1 namespace alive". The probe should call this exactly once and treat 200 vs 401/403 vs 404 as a tristate.

[source: https://github.com/sonatype-nexus-community/nexus-repo-api-client-go/blob/main/docs/StatusAPI.md]

### 1.4 Other candidates evaluated

| Candidate | Verdict | Why |
|---|---|---|
| `/service/rest/v1/status` | Keep | Already in probe v0; works without auth on most installs but doesn't return a version body |
| `/service/rest/v1/status/writable` | Keep | Same — narrower (read-write probe), 200 vs 503 |
| `/service/rest/atlas/system-information` | Drop for non-admin | Admin-only (`nx-atlas-all` privilege) [source: support.sonatype.com nx-metrics-all article] |
| `/service/rest/internal/ui/version` | Inferred not useful | Not in the OpenAPI catalog; UI-internal endpoints are unstable across versions |
| `/service/rest/swagger.json` | Useful for **capability discovery**, not version | Documented as "does not require any privilege to access" [source: https://help.sonatype.com/en/api-reference.html — "The Open API Document at /service/rest/swagger.json … does not require any privilege to access it"]. Body contains `"version": "..."` in the `info` block — so technically yes, but it's a 200 KB+ JSON to parse for a single string |
| `/service/rest/v1/system/license` | Pro-only, admin-likely | Returns `ApiLicenseDetailsXO` — useful but probably 403 for a non-admin token; probe and observe |
| `GET /service/rest/v1/system/node` | Probably non-admin | Returns `NodeInformation` (Pro/HA only); worth a probe |
| Rapture cache-buster HTML scrape | Keep as final fallback | Documented hack; works without any auth at all [source: probe v0] |

### 1.5 Recommended detection ladder (probe v0.1)

```
1. Server: header on the very first /v2/ or /service/rest/v1/status call → primary
2. /service/rest/v1/status/check body → confirms namespace alive AND version-detect via Server: header on that response
3. swagger.json → only if 1+2 fail; parse JSON .info.version
4. Rapture HTML scrape on / → final fallback for paranoid reverse-proxy strip cases
```

**Probe note:** the OCI manifest accept-header dance is THE big swap for the plugin. Today the plugin requests v0's default media type, which on a multi-arch image returns the manifest-list, but the plugin treats it as a v2 manifest and likely fails downstream. The probe should issue four parallel HEAD requests with each Accept variant and report which the registry uses for the first tag of the first repo.

## 6. Recommended probe v1 design changes

1. **Drop `/service/rest/v1/system/about`** — phantom on 3.90, never existed in `/v1/`.
2. **Capture `Server:` response header on every raw/*.json save** — primary version-detect signal. Parse `Nexus/(\d+\.\d+\.\d+)-(\d+)\s*\((\w+)\)` for version + build + edition.
3. **Add `/service/rest/v1/status/check`** as secondary version-detect. Treat tristate: 200 (namespace + auth ok), 401/403 (auth or perm fail), 404 (pre-3.81 or namespace gone).
4. **Add `/service/rest/swagger.json`** as `version` AND capability-discovery — parse `info.version` (string) + `paths` (object keys → list of available endpoints). The probe should diff "what spec advertises" vs "what we successfully called".
5. **Keep rapture `?3.90.1-01` HTML scrape** as final fallback only.
6. **Add `--format-repo <name>`** arg (or auto-detect first maven repo from `/v1/repositories`) to enable `§3.6` per-format probes (`maven-metadata.xml`, `.sha256` sidecar, `HEAD /repository/...`).
7. **Add OCI manifest media-type variants** under `/v2/{repo}/manifests/{tag}` — issue four parallel HEAD requests with each Accept header and record which the registry returns. Single highest-value swap finding for the plugin.
8. **Add `Link:` header capture** on `/v2/_catalog` and `/v2/{repo}/tags/list` so the redact bundle records pagination metadata, not just first-page content.
9. **Add `--registry-test-blob-head`** that does `HEAD /v2/{repo}/blobs/{digest}` for the digest from the first manifest — confirms blob layer existence cheaply without downloading.
10. **Consolidate the existing capability-marker calls** (`/v1/security/users`, `/v1/security/realms/active`, `/v1/tasks`, `/v1/blobstores`, `/v1/system/license`) under one `--capability-probe` flag. They all serve the same purpose: "what can this token actually do?". Output a single matrix.
11. **3.88+ wildcard fix**: probe `?q=neo*` AND `?q=*neo*` in two calls and record which Nexus accepts — confirms whether the instance is on the SQL search path or still pre-3.88. The plugin's eventual UI autocomplete must respect "trailing wildcards only".
12. **3.90 Recovery Mode probe**: `GET /v1/recovery-mode` to detect if the instance is in recovery mode (would explain weird search/upload behavior). Newly-added in 3.90.0.

## 7. Open questions / things the response payloads should answer

The probe full sweep is the cheapest way to answer these — none can be answered statically.

1. **Which formats are present?** Inspect `/v1/repositories` response → `format` field per repo. Likely `maven2 + docker`, but possibly `npm`, `raw`, `nuget`, `pypi`, `helm`. Each new format opens a §3.6 probe.
2. **Is HA enabled?** `/v1/system/node` returns NodeInformation only on Pro+HA. 200 vs 404 vs 403 answers it.
3. **Is replication / staging / smart-proxy configured?** `/v1/replication/*` and `/v1/staging/*` return 404 if not configured. Probe and capture.
4. **Are custom cleanup policies / routing rules / content selectors deployed?** Three GETs answer it. If 403, not visible to this token; if 200 with empty array, none configured; if 200 with entries, we have data to drive UI features.
5. **Is the IQ link configured?** `/v1/firewall/configuration` returns 200 with config or 404/disabled. Tells us "is policy-driven artifact blocking active in this Nexus".
6. **Is anonymous access on?** `/v1/security/anonymous` (admin) or **infer from response headers** — if `/v2/` or `/v1/repositories` returns 200 with no `Authorization:` header, anon access is on for that path. Probe should make ONE unauth call to `/v1/repositories` and record the status code.
7. **Multi-arch Docker images?** Inspect first manifest with all 4 Accept headers. If the instance returns OCI image-index, **the plugin's current single-manifest assumption is broken for multi-arch images** — high-priority swap.
8. **Is `swagger.json` reachable?** If yes, the probe gets a free machine-readable inventory of EVERY endpoint this instance ships, which is the most-comprehensive capability-discovery signal possible.
9. **Server header presence?** Confirm `Server: Nexus/...` is not stripped by reverse proxy. If stripped on the admin host but present on the docker host (or vice versa), document it — affects version-detect ladder ordering.
10. **Recovery mode active?** `/v1/recovery-mode` GET — if the response body indicates active, search/upload flows will behave unusually and the plugin should surface a banner.

## Appendix B — 3.90-specific delta from prior 3.x

Confirmed against [3.90.0 release notes](https://help.sonatype.com/en/sonatype-nexus-repository-3-90-0-release-notes.html):

- **NEW**: User Token API admin endpoints — `GET/POST/DELETE /v1/security/user-tokens/tokens`. Probe expects 403 for non-admin.
- **NEW**: Recovery Mode API — `GET/POST/DELETE /v1/recovery-mode`. Admin-only.
- **NEW**: Log Management `nexus:logging:read` permission — non-admin tokens with this priv can pull logs (was `nx-admin`-only previously).
- **CHANGED**: Docker repos must be lowercase. Probe lowercases repo names before /v2/ calls.
- **REMOVED**: Legacy "Application Health Check" + "Hosted Repository Analysis" plugins. The `/v1/repositories/{name}/health-check` body content is **smaller in 3.90 than in 3.85**. Probe should not assume specific fields beyond what the current help page documents.
- **CHANGED**: npm metadata exposes `sonatype_filtered_versions` — no API path change, but raw package metadata at `/repository/{npm-repo}/{pkg}` will show this field.
- **DEPRECATED-LATER**: Docker image base shifts to Alpine in 3.91 with suffix deprecation in 3.94 — affects how the user runs Nexus, not the probe.
- **From 3.81**: `/service/metrics/*` → `/service/rest/metrics/*` and `/service/metrics/healthcheck` → `/service/rest/v1/status/check`. Redirects exist but probes should call new paths directly.
- **From 3.88**: SQL search replaced Elasticsearch — `q=*foo` and `q=f*o` no longer work, only `q=foo*`.
