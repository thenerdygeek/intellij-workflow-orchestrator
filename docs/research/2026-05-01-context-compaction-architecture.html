1.2 Two layers of defense

Layer 1 — ToolOutputSpiller (upstream)

Per-tool output cap (50K default, 100K for run_command). Outputs >30K go to disk; only head-20 + tail-10 preview enters the message list. Runs before ContextManager sees anything.

Layer 2 — ContextManager (downstream)

Handles accumulated message bloat — many small tool results, repeated file reads, long conversation history. Runs on the message list as a whole.

★ Why two layers? Layer 1 prevents a single 5MB stack trace from blowing the context in one shot. Layer 2 prevents a thousand 5KB tool results from doing the same thing slowly. Independent state — they don't share counters or caches.

3.2 Stage 1 — Duplicate file-read dedup

Tracking discipline. fileReadIndices is populated in addToolResult() by inspecting the prior assistant message's tool_calls, recognizing tools in FILE_READ_TOOLS = {read_file, create_file, edit_file}, and parsing the path from the result body (header-style or tag-style).

If savedChars / totalChars ≥ 30%, the pipeline exits — Stage 2/3 don't run. Most compactions on long sessions stop here.

Three boundary-correctness rules (load-bearing)
  1. Always end on assistant. If messages[rangeEnd].role != "assistant", decrement so user-asst alternation survives the cut.
  2. Absorb orphan tool results. If the next surviving message is tool, walk forward absorbing all consecutive tool messages — otherwise the surviving tool result has no preceding assistant.tool_calls to pair with, breaking OpenAI-format validation.
  3. Notice replaces index 1 (not inserted). Avoids shifting all later indices and preserves the "first user-assistant pair" anchor as exactly indices 0 and 1.

★ Three load-bearing decisions.
  1. Role is assistant, not user. The next surviving message is also user, so consecutive user roles would break provider validation.
  2. Summary chaining — each successive Stage 3 includes the prior lastSummary, so memory accumulates across compactions.
  3. Failure is silent. If the API errors, no mutation happens. No partial state.

★ Two-file persistence design. The user keeps seeing the full chat scrollback even after the LLM-facing context has been compacted. This is great UX — but it means there's no audit trail of "what the LLM saw at iteration N" once onHistoryOverwrite runs. Roo Code's condenseParent tag pattern (mark-don't-delete) would solve this.

8. Where We Sit on the Cost ↔ Coherence Spectrum

quadrantChart title Cost vs Coherence — 9 agents x-axis "Low cost (no LLM)" --> "High cost (every compaction = LLM)" y-axis "Low coherence (drop)" --> "High coherence (preserve narrative)" quadrant-1 Best of both quadrant-2 Coherent, expensive quadrant-3 Cheap, lossy quadrant-4 Expensive, lossy Continue: [0.05, 0.15] Cline: [0.1, 0.45] "This plugin": [0.25, 0.6] OpenCode: [0.6, 0.75] "Claude Code": [0.7, 0.85] Aider: [0.65, 0.55] "Codex CLI": [0.7, 0.6] "Roo Code": [0.75, 0.7] Plandex: [0.55, 0.5]
★ Our design space. Closest to OpenCode in shape (dedup + summary + replay), but sit further left on cost because Stage 3 (LLM summary) only fires above 95% — most compactions stop at Stage 1 or 2 and pay zero LLM cost. The trade-off: we don't have OpenCode's structured Markdown summary template (no schema validation), and our explicit recency-preservation block is implicit in the truncation math rather than a crisp DEFAULT_TAIL_TURNS=2 constant.

9. Gaps — Ranked by Impact

9.1 High-impact

G1No semantic dedup beyond read_file. Stage 1 only catches 3 file-read tools. Identical results from search_code, find_definition, find_references, db_schema, glob_files accumulate as duplicates. Fix: hash-based dedup keyed on (toolName, normalizedArgs) — extends dedup to ~30 tools. OpenCode does this for any deterministic tool.
G2Stage 3 prompt produces unparsed string. TASK/FILES/DONE/ERRORS/PENDING is "trust the LLM" — no validation. OpenCode's SUMMARY_TEMPLATE with explicit "every section, even when empty" rule is more robust. Schema-validated tool-call would let us reject malformed summaries.
G3No "summaries-of-summaries" depth cap. lastSummary chains forever. Codex's Rust rewrite explicitly fixed this by using a fresh template each pass. Pathological case: 10+ depth dilutes early information.
G4No session-wide circuit breaker. We have per-iteration retry counters but no kill-switch if compaction repeatedly fails to shrink the prompt enough. Codex bug #13946 hit this: infinite loop when summary produced is still too large. Add a session-wide "disable compaction after N consecutive ineffective compactions."

9.2 Medium-impact

G5compactionThreshold hardcoded. Roo: 5–100%. Claude Code: env override. We: 0.85 in constructor. Adding agentSettings.compactionThreshold would help advanced workflows.
G6No "protected tools" allowlist. OpenCode protects ["skill"] from output pruning. Our skill survives via re-injection (belt-and-suspenders) but mid-conversation skill-load tool result isn't tagged-as-protected.
G7No cost-aware Stage 3 trigger. Fires whenever util > 95% regardless of whether convo is mostly tool noise (where summary wastes a call) or substantive turns (where it preserves value). Heuristic: if tool_results_chars / total_chars > 0.7, prefer aggressive truncation.
G8No audit trail of pre-compaction history. onHistoryOverwrite is destructive. Roo's condenseParent tag pattern (non-destructive) is better — tag-don't-delete, hide from API view but keep on disk for debugging.

9.3 Low-impact polish

G9No model-tier-aware reserve buffer. Cline scales 27K/30K/40K by tier. We use one 190K. On a 1M-context model the same 85% means compacting at 850K — possibly too late if next response wants 32K output.
G10LAST_TWO and NONE strategies coded but unreachable. Presumably for the not-yet-wired manual "compact aggressively" UI button.
G11PRE_COMPACT hook telemetry sparse. Currently passes {utilizationPercent, messageCount}. Adding lastPromptTokens, fileReadIndices.size, toolDefinitionTokens would let users write smarter hooks.
